As you see from the mock-up below sthe economic operator can add and remove technicians and bodies associated to one CA's REQUIREMENT. In this case the REQUIREMENT specified by the CA is the type of technical assistance the EO's teams must provide.

==== Data Structure - Self-contained

In this case the data structure is a group of one REQUIREMENT that has associated two sub-groups of QUESTIONs. It actually does not matter whether the groups and sub-groups used to organise the CAPTIONS, REQUIREMENTS and QUESTIONS are declared as REQUIREMENT_GROUP, REQUIREMENT_SUBGROUP or QUESTION_GROUP and QUESTION_SUBGROUP, as all of them will end up transformed into `cac:TenderingCriterionPropertyGroup` and `cac:SubsidiaryTenderingCriterionPropertyGroup` elements. In this case the author of the structure has considered that there was a REQUIREMENT that is linked to a set of sub-groups of QUESTIONS, this is why the outer level has been declared as a REQUIREMENT_GROUP (but, again, the result would have been the same if the outer group had been declared as a QUESTION_GROUP).

The important aspect here is that the sub-groups of QUESTIONs, the one for the technicians and the one for the technical bodies, can be repeated several times, one time per each technician or body the economic operator decides to add (thus the cardinalities 0..n for both sub-groups).

==== XML Example - Self-contained

Observe that the contracting authority (CA) can associate one REQUIREMENT to one group of several technicians and technical bodies. In this case the requirement is the type of technical support it expects from the economic operator (EO). Per each technician and technical body about which the economic operator (EO) supplies data new subgroups of properties are dynamically created. Thus, in the figure below you can see that the economic operator has created three sub-groups of properties for three different technicians and two sub-groups to place the data about the technical bodies (the nodes are collapsed to clarify the structure, but you can find the complete XML example below). See also the corresponding XML file in the distribution package link:./dist/xml[XML examples folder]. If you do not find the XML example you can take the link:./dist/cl/ods/ESPDRequest-CriteriaTaxonomy-SELFCONTAINED-V02.00.00.ods[spread-sheet] containing the data structures and generate the XML sample using the self-contained transformation link:./dist/xslt/SELFCONTAINED-ESPDRequest-Annotated-V02.00.06.xslt[XSL style-sheet].
